# AI_LOG.md — 第一題 資料清理 (Data Cleaning)

## 學號專屬參數
- 整除數 D = 3

## 開工前五項檢查表

### ① 函式簽名 (Function Signature)
- 採用 `sys.stdin.read()` 一次性讀取全部輸入，再以 split 後的 token 串流逐一解析。
- 主處理邏輯規劃為單一函式：`process(arr: list[int], d: int) -> list[int]`，輸入一組整數列表與整除數，回傳「去重→篩選→排序」後的結果列表（不含輸出格式化）。
- 輸出格式化（含 NONE 判斷）獨立於函式外的主迴圈處理，不混入核心邏輯，方便單獨測試。

### ② 輸入邊界
- n 的範圍：1 ≤ n ≤ 10^5，多組測資需迴圈讀取直到 n = 0。
- 數值範圍：−10^9 ≤ a_i ≤ 10^9（Python int 原生支援，無溢位問題，但要留意負數的整除判斷）。
- 終止條件 n = 0 時，該組「不處理、不輸出」，且必須正確跳出迴圈，不可誤讀下一行。

### ③ 例外處理
- 若某組篩選後結果為空列表，必須輸出大寫字串 `NONE`，不可輸出空白行。
- 整除判斷必須用 `ai % D == 0`，需驗證此寫法對負數與 0 是否仍正確（Python 的 `%` 對負數行為與數學定義一致，需特別注意 0 % D == 0 永真）。
- 讀取資料時最後一組 n=0 後若仍有殘留 token，不應再嘗試解析。

### ④ Edge Case 設計重點
- 重複值中包含負數與 0，驗證去重保留「第一次出現順序」是否正確。
- 篩選後結果為空 → 必須輸出 `NONE`。
- 排序需確認負數、0、正數混合時的由小到大排序正確。

### ⑤ 驗收標準
- 兩組學校 Sample Input 套用 D=3 後，輸出必須與下方「紅燈測試案例」逐字元一致。
- 邊界案例（含負數、0、重複值）需正確處理，不可拋出例外或輸出格式錯誤。
- 程式需能處理 n 高達 10^5 的情況不超時（避免 O(n²) 去重或篩選）。

## 🔴 紅燈測試案例（Red Light Test Cases）

### 測資 1（學校 Sample，D=3）
**Input：**
```
8
4 7 4 2 9 2 6 7
```
**處理過程：**
- 去重保序：`4, 7, 2, 9, 6`
- 篩選 D=3 整除：`9, 6`
- 排序：`6, 9`

**Expected Output：**
```
6 9
```

### 測資 2（學校 Sample，D=3）
**Input：**
```
3
1 3 5
```
**處理過程：**
- 去重保序：`1, 3, 5`
- 篩選 D=3 整除：`3`
- 排序：`3`

**Expected Output：**
```
3
```

### 🧩 邊界案例（Edge Case，自行設計）
**設計動機：** 同時驗證「負數整除判斷」、「0 的整除永真特性」、「負數排序」、「重複值含負數的去重保序」四個風險點。

**Input：**
```
6
-3 -3 9 9 -9 0
```
**處理過程：**
- 去重保序：`-3, 9, -9, 0`
- 篩選 D=3 整除（-3, 9, -9, 0 皆可被 3 整除）：`-3, 9, -9, 0`
- 排序（由小到大）：`-9, -3, 0, 9`

**Expected Output：**
```
-9 -3 0 9
```
